# ADR 0010 — TPM-backed boot-state attestation

Status: accepted (2026-05-15)

## Context

Phase 4 #4.10 added TPM-backed attestation of agent boot state. The
goal is to detect tampering between agent installs: a host that
arrived in the fleet with PCR values matching a trusted reference set
should not, over time, drift into a different boot configuration
without the operator noticing. Common drift vectors the feature
should catch: unauthorised SecureBoot disable, a malicious bootloader
swap, an early-boot kernel module insertion, a UEFI firmware
implant.

The pre-Phase-4 agent reported a heartbeat saying "I am alive"; it
did not report any cryptographic evidence of *what kind of system it
is alive on*. An attacker with persistence at the bootloader or
firmware level could disable agent self-protection, replay heartbeats,
or exfiltrate certificates without changing any signal the manager
already collected.

Two shapes were on the table:

1. **Software measurement.** Hash a set of files at agent boot
   (kernel image, initramfs, agent binary) and compare to a known-
   good. Trivially defeated by an attacker who has already
   compromised those files — the attacker simply hashes the
   original.
2. **TPM 2.0 Quote.** The TPM holds PCRs that the firmware extended
   during the actual boot sequence. The agent asks the TPM to sign a
   quote over those PCRs with an attestation key the manager
   pre-registered. An attacker who tampers with boot changes the PCR
   values; the TPM will not lie about them.

## Decision

Agent attestation via TPM 2.0 PCR Quote, with the manager as the
verifier.

- **PCRs in scope: 0, 1, 2, 3, 4, 5, 7, 14.** PCR 0-5 cover firmware,
  option ROMs, MBR / IPL; PCR 7 covers SecureBoot state and
  authorities; PCR 14 (Linux IMA / Windows BCD) covers OS-level
  measurements. The set is configurable per-host-group via
  `host_group.attestation_pcr_mask` for operators with non-default
  measurement policies.
- **Attestation Key (AK) is endorsed by the agent's TPM EK.** First
  install: the agent generates an AK in the TPM, certifies it with
  the EK, and sends `EKpub + AKpub + EKcert + AKcert` to the manager
  during enrollment. The manager validates the EK cert against the
  vendor CA bundle baked into the manager image
  (`backend/app/services/tpm_ek_trust.py`).
- **Reference PCR set is captured at first enrollment.** The agent's
  first successful quote becomes the reference. Subsequent quotes are
  compared bit-for-bit; a mismatch opens an alert with rule kind
  `attestation_drift` at HIGH severity. Operators can re-baseline
  via the host detail page (audited as
  `host.attestation.rebaseline`).
- **Quote cadence: every heartbeat, lazily.** The manager piggybacks
  a `please_quote` flag on the heartbeat response when the last
  quote is older than `VIGIL_ATTEST_INTERVAL_S` (default 3600 s).
  The agent's next heartbeat carries the quote.
- **Quote verification on the manager.** The manager validates: AK
  signature over the quote, nonce freshness (per-quote random 16
  bytes generated by the manager and embedded in the
  `please_quote`), PCR values against reference. Verification lives
  in `backend/app/services/tpm_quote_verifier.py`.
- **TPM unavailable is a known status, not an enrollment block.**
  Hosts without a TPM (VMs without vTPM, older hardware) enroll
  successfully and report `attestation_status = unavailable`.
  Operators see the status in the host list and choose whether to
  exclude such hosts from sensitive host groups.

## Why not Windows Device Health Attestation (DHA) / Linux IMA alone

- **DHA is Microsoft-cloud-hosted** and would force a hard
  dependency on Microsoft's attestation service. The product targets
  air-gapped operators; we cannot require their hosts to call out.
  We implement the underlying flow ourselves.
- **Linux IMA covers PCR 10 only** and requires the operator to have
  signed every executable they want measured. Many operators do not.
  We accept IMA on the boxes where it is configured (operators surface
  the PCR via `attestation_pcr_mask`) without requiring it everywhere.

## Failure modes the manager must tolerate

- **TPM clears.** A genuine firmware update can re-extend PCRs; the
  agent will quote and the verification will mismatch. Operators
  acknowledge the alert and rebaseline; the audit row records who
  rebaselined and when.
- **EK cert vendor not in our trust bundle.** Enrollment proceeds
  with `attestation_status = ek_untrusted`; operators add the vendor
  bundle via the host-group attestation settings or accept that the
  host attests against itself only (TOFU mode).
- **Replay of a stale quote.** The per-quote nonce prevents this; a
  quote without a matching outstanding nonce is rejected and audited
  as `host.attestation.replay_attempt`.

## Consequences

- The agent gains a TPM 2.0 dependency on Linux (via
  `tss2-esys`) and Windows (via `tbs.dll`). The agent ships these
  bindings statically where possible.
- The enrollment protocol grew an `attestation` substream in the
  gRPC service (`AgentService.Attest`). See ADR 0002.
- The host model carries: `attestation_status`, `ek_cert_thumbprint`,
  `ak_pub`, `pcr_reference` (jsonb), `last_quote_at`, `last_drift_at`.
- New rule kind `attestation_drift` in the alert pipeline. Routed
  through the normal Sigma realtime pipeline but matched against
  the structured quote-mismatch event rather than ECS log shapes.
- Migration `f6f7a8b9c0d1` (the chain base referenced in ADR 0007)
  introduced these columns.

## Outstanding follow-ups

- Remote attestation of the manager *itself* — useful for an
  operator who wants to prove to their auditor that the manager has
  not been tampered with.
- TPM 1.2 fallback. Out of scope; modern hardware ships 2.0.
- Cross-vendor EK cert bundle refresh automation — currently the
  bundle is updated in the manager image; a periodic refresh job
  would let operators pick up new vendor CAs without a redeploy.
